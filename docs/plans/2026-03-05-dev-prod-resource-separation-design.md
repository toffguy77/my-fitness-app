# Dev/Prod Resource Separation — Design

## Overview

Разделение шареных ресурсов Yandex Cloud на отдельные dev и prod окружения.
Подход: ручное создание через `yc` CLI с документированием команд в runbook.

## Scope

Разделяем:
- PostgreSQL (две БД в одном кластере)
- S3 (префиксы `dev/` и `prod/` в существующих бакетах)
- Сервисные аккаунты (отдельные SA для dev и prod)

Не в скоупе: SMTP.

## 1. PostgreSQL — две БД в одном кластере

**Текущее состояние:** одна БД `web-app-db` в кластере `c-c9q1384cb8tqrg09o8n4`, один пользователь.

**Целевое состояние:**
- **prod БД:** `web-app-db` (существующая, без изменений)
- **dev БД:** `web-app-db-dev` (новая, в том же кластере)
- **prod пользователь:** существующий, доступ только к `web-app-db`
- **dev пользователь:** новый `burcev-dev`, доступ только к `web-app-db-dev`
- Миграции применяются к обеим БД независимо через разные `DATABASE_URL` в env-файлах

**Изменения в коде:** нет — приложение берёт `DATABASE_URL` из env.

## 2. S3 — префиксы в существующих бакетах

**Текущее состояние:** 4 бакета (weekly-progress-photos, profile photos, chat, content) без разделения.

**Целевое состояние:**
- Те же 4 бакета
- **Dev** пишет с префиксом `dev/`
- **Prod** пишет с префиксом `prod/`
- Новая env-переменная `S3_PATH_PREFIX` (`dev/` для dev, `prod/` для prod)

**Изменения в коде:**
- В Go S3-клиенте (`internal/shared/storage/`): при формировании ключа объекта добавлять `S3_PATH_PREFIX` перед путём
- Затрагивает все места формирования S3 key

**Примечание:** реальных prod-пользователей пока нет, поэтому добавление `prod/` префикса не ломает существующие данные.

## 3. Сервисные аккаунты — отдельные SA

**Текущее состояние:** один SA `ajetieia8uunpq733f9t` с доступом ко всем ресурсам.

**Целевое состояние:**
- **SA `burcev-prod`** — доступ: БД `web-app-db`, все 4 S3 бакета
- **SA `burcev-dev`** — доступ: БД `web-app-db-dev`, те же S3 бакеты
- Для каждого SA: отдельные static access keys (S3) и authorized keys
- Разделение S3 по prefix — на уровне приложения, не IAM
- Старый SA деактивируется после перехода

**Изменения в коде:** нет — SA credentials берутся из env.

## 4. Env-файлы

**Обновляются:**
- `deploy/env/.env.dev.example` — `DATABASE_URL` (dev БД, dev user), `S3_PATH_PREFIX=dev/`, новые SA ключи
- `deploy/env/.env.prod.example` — `DATABASE_URL` (prod БД, prod user), `S3_PATH_PREFIX=prod/`, новые SA ключи
- На сервере: `.env.dev` и `.env.prod` с реальными значениями

## Инструментарий

- **Terraform** для управления облачными ресурсами (IaC)
  - Расположение: `infra/` в корне монорепо
  - Разделение окружений: `dev.tfvars` / `prod.tfvars`
  - State: Yandex Cloud S3 bucket
  - Существующие ресурсы (текущий SA, текущая БД, кластер) импортируются в state
- Провайдер: `yandex-cloud/yandex`
