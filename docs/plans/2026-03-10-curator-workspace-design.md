# Curator Workspace Design

Date: 2026-03-10

## Summary

Comprehensive redesign of the curator workspace using a "Hub + Detail" architecture. The curator gets an analytics hub for portfolio overview and an expanded client card with tabs for plans, tasks, and feedback. Client dashboard is updated to surface curator-assigned plans, tasks, and weekly feedback.

## Requirements

- **Approach:** Hybrid — active coaching for new clients + monitoring for established
- **Plans:** Auto-calculated KBZHU as base + curator override per field
- **Tasks:** Structured with types (nutrition/workout/habit/measurement) and recurrence (once/daily/weekly)
- **Feedback:** Structured by categories (nutrition/activity/water/photo) + summary + recommendations
- **Analytics:** Full dashboard with daily/weekly snapshots, platform benchmark + own history comparison
- **Grouping:** Strictly individual work, no group tools

## Architecture: Hub + Detail

Three levels:
1. **Curator Hub** (`/curator`) — analytics + priority list + client list
2. **Client Card** (`/curator/clients/:id`) — tabs: Overview, Plan, Tasks, Reports
3. **Existing screens** — chat and content remain separate sections

---

## Section 1: Analytics Hub (`/curator`)

Replaces the current simple client list. Becomes the curator's main page.

### Top Block — Summary Metrics (4 cards)

| Metric | Description |
|--------|-------------|
| **Active clients** | Total count, of which "need attention" (red/yellow alerts) |
| **Avg KBZHU completion %** | Current week across all clients (% of calorie plan) |
| **Unread messages** | Total count + number of clients with unread |
| **Tasks under control** | Overdue + completed today out of total active |

### Middle Block — Priority List "Need Attention"

Auto-generated by priority:
1. **Red alerts** — calories <50% or >120% of plan
2. **Overdue tasks** — client missed deadline
3. **No activity 2+ days** — no food entries
4. **Unread messages** — awaiting curator response
5. **Awaiting feedback** — weekly report submitted, curator hasn't responded

Each item: compact card with alert reason + quick action (go to card / open chat).

### Bottom Block — Full Client List

Current `ClientList` enhanced with:
- Filters: all / need attention / on track
- Sorting: by priority (default) / by name / by assignment date
- Each card: mini-bar showing weekly KBZHU completion % (instead of today-only)

## Section 2: Expanded Client Card (`/curator/clients/:id`)

### Tab Navigation

| Tab | Content |
|-----|---------|
| **Overview** (default) | Current content: days with KBZHU, weight, water, steps, workouts, photos |
| **Plan** | Weekly nutrition plan + plan history |
| **Tasks** | Client task list + creation |
| **Reports** | Weekly reports + curator feedback |

### Tab "Overview" — Changes

Existing content remains (`DaySection`, `KBZHUProgress`, weight chart, photo gallery). Additions:
- **Mini-summary at top** — client status: consecutive diary days (streak), weight trend for month (up/down/stable with kg delta)
- **Quick actions** — buttons: "Message in chat", "Adjust plan", "Add task" (navigate to respective tabs/chat)

### Tab "Plan"

**Current plan** (if active `weekly_plan` exists):
- Period: start date — end date
- Target values: calories, protein, fat, carbs
- Button "Adjust"

**Create/edit plan:**
- System shows **auto-calculated** KBZHU values (from client profile) as base
- Curator can override any field (calories, P, F, C independently)
- On calorie change — offers to recalculate macros proportionally (optional)
- Period: start + end (default Monday-Sunday)
- Curator comment (optional, visible to client)

**Plan history:**
- List of past plans with dates and values
- Click to expand details

**Backend:** CRUD endpoints `POST/PUT/DELETE /api/v1/curator/clients/:id/weekly-plan` in `curator` module. Table `weekly_plans` already exists.

### Tab "Tasks"

**Task list:**
- Filters: active / completed / overdue
- Each task: name, type, deadline, status, recurrence

**Create task:**
- **Name** — free text
- **Type** — nutrition / workout / habit / measurement
- **Description** — optional, extended text
- **Deadline** — date
- **Recurrence** — once / daily / weekly
- For recurring: weekdays or interval

**Progress:**
- Client marks completion in their dashboard
- Curator sees status: completed / not completed / overdue
- For recurring: mini-calendar (7 days, filled cells)

**Backend:** CRUD endpoints `/api/v1/curator/clients/:id/tasks`. Table `tasks` already exists in DB.

### Tab "Reports"

**Weekly report list:**
- Auto-generated on Sundays (as now)
- Status: "Awaiting feedback" / "Feedback given"

**Report view:**
- Auto-collected data: avg calories, avg weight, steps, workouts, week photos
- Task progress for the week

**Structured curator feedback:**

| Category | Rating | Comment |
|----------|--------|---------|
| Nutrition | excellent / good / needs_improvement | Text |
| Activity | same | Text |
| Water | same | Text |
| Photo | uploaded / not_uploaded | — |

Plus:
- **Summary** — free text, week summary
- **Recommendations for next week** — text, visible to client in dashboard

**Backend:** `PUT /api/v1/curator/clients/:id/weekly-reports/:id/feedback`. Column `curator_feedback` in `weekly_reports` expanded to JSON structure.

## Section 3: Client-Side Changes

### Client Dashboard (`/dashboard`)

**"Weekly Plan" section** (exists but empty — no data):
- Shows current plan from curator: target KBZHU + curator comment
- Visual comparison: this week's plan vs last week's (up/down per macro)

**"Tasks" section** (exists but empty):
- List of active tasks for current week/day
- Checkbox for marking completion
- Task type icon (nutrition, workout, habit, measurement — via Lucide icons)
- Recurring tasks: mini progress bar (4/7 days completed)
- Overdue tasks highlighted

**"Curator Feedback" section** (new):
- Appears after Sunday report when curator leaves feedback
- Card with category ratings (color indicators: green/yellow/red)
- Summary and recommendations
- Collapsible, remains visible until end of next week

### New Client Notification Types

| Event | Notification |
|-------|-------------|
| Curator created/changed plan | "Your curator updated the nutrition plan for this week" |
| Curator assigned task | "New task: {name}" |
| Curator gave feedback | "Curator left feedback on your report" |

All notifications clickable — navigate to corresponding dashboard section.

### Weekly Report Submission

Current mechanism (auto-collect on Sundays) enhanced:
- Status indicator: "Report sent, awaiting feedback" / "Curator left feedback"
- Client sees their report + curator feedback in one place

## Section 4: Backend — New Endpoints and DB Changes

### New API Endpoints

**Curator module:**

```
# Weekly plans
POST   /api/v1/curator/clients/:id/weekly-plan
PUT    /api/v1/curator/clients/:id/weekly-plan/:planId
DELETE /api/v1/curator/clients/:id/weekly-plan/:planId
GET    /api/v1/curator/clients/:id/weekly-plans

# Tasks
POST   /api/v1/curator/clients/:id/tasks
PUT    /api/v1/curator/clients/:id/tasks/:taskId
DELETE /api/v1/curator/clients/:id/tasks/:taskId
GET    /api/v1/curator/clients/:id/tasks              (?status=active|completed|overdue)

# Feedback
PUT    /api/v1/curator/clients/:id/weekly-reports/:reportId/feedback
GET    /api/v1/curator/clients/:id/weekly-reports

# Hub analytics
GET    /api/v1/curator/analytics
GET    /api/v1/curator/attention

# Analytics history & benchmarks
GET    /api/v1/curator/analytics/history    (?period=daily&days=30 | ?period=weekly&weeks=12)
GET    /api/v1/curator/analytics/benchmark  (?weeks=12)
```

**Dashboard module (client endpoints):**

```
GET    /api/v1/dashboard/tasks
PUT    /api/v1/dashboard/tasks/:taskId/complete
GET    /api/v1/dashboard/weekly-reports/:reportId/feedback
```

### DB Changes

**Table `tasks`** (exists, extend):

```sql
ALTER TABLE tasks ADD COLUMN IF NOT EXISTS type VARCHAR(20) DEFAULT 'habit';
  -- types: nutrition, workout, habit, measurement
ALTER TABLE tasks ADD COLUMN IF NOT EXISTS recurrence VARCHAR(20) DEFAULT 'once';
  -- once, daily, weekly
ALTER TABLE tasks ADD COLUMN IF NOT EXISTS recurrence_days INTEGER[];
  -- for weekly: [1,3,5] = Mon,Wed,Fri
```

**New table `task_completions`:**

```sql
CREATE TABLE task_completions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  completed_date DATE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(task_id, completed_date)
);
```

**Table `weekly_reports`** (exists, expand `curator_feedback`):

```sql
ALTER TABLE weekly_reports
  ALTER COLUMN curator_feedback TYPE JSONB USING curator_feedback::jsonb;
```

Feedback JSON structure:
```json
{
  "nutrition":  { "rating": "good", "comment": "..." },
  "activity":   { "rating": "needs_improvement", "comment": "..." },
  "water":      { "rating": "excellent", "comment": "..." },
  "photo":      { "uploaded": true },
  "summary":    "General comment",
  "recommendations": "Recommendations for next week"
}
```

Ratings: `excellent` / `good` / `needs_improvement`.

**New table `curator_daily_snapshots`:**

```sql
CREATE TABLE curator_daily_snapshots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  curator_id UUID NOT NULL REFERENCES users(id),
  date DATE NOT NULL,
  total_clients INTEGER,
  attention_clients INTEGER,
  avg_kbzhu_percent NUMERIC(5,1),
  total_unread INTEGER,
  active_tasks INTEGER,
  overdue_tasks INTEGER,
  completed_tasks INTEGER,
  avg_client_streak NUMERIC(5,1),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(curator_id, date)
);
```

**New table `curator_weekly_snapshots`:**

```sql
CREATE TABLE curator_weekly_snapshots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  curator_id UUID NOT NULL REFERENCES users(id),
  week_start DATE NOT NULL,
  avg_kbzhu_percent NUMERIC(5,1),
  avg_response_time_hours NUMERIC(5,1),
  clients_with_feedback INTEGER,
  clients_total INTEGER,
  task_completion_rate NUMERIC(5,1),
  clients_on_track INTEGER,
  clients_off_track INTEGER,
  avg_client_streak NUMERIC(5,1),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(curator_id, week_start)
);
```

**New table `platform_weekly_benchmarks`:**

```sql
CREATE TABLE platform_weekly_benchmarks (
  week_start DATE PRIMARY KEY,
  avg_kbzhu_percent NUMERIC(5,1),
  avg_response_time_hours NUMERIC(5,1),
  avg_task_completion_rate NUMERIC(5,1),
  avg_feedback_rate NUMERIC(5,1),
  avg_client_streak NUMERIC(5,1),
  curator_count INTEGER,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**Snapshot collection:**
- Daily — cron job or Go ticker, runs nightly
- Weekly — Mondays, aggregates past week data
- Platform benchmarks — computed simultaneously with weekly snapshots (average across all curators)

### New Notification Types

- `plan_updated` — curator updated plan
- `task_assigned` — task assigned
- `task_overdue` — task overdue
- `feedback_received` — curator gave feedback

Delivered via existing WebSocket channel.

## Section 5: Analytics Details

### Priority List (`GET /api/v1/curator/attention`)

Returns sorted array:
```json
{
  "client_id": "uuid",
  "client_name": "string",
  "client_avatar": "url",
  "reason": "red_alert | overdue_task | inactive | unread_message | awaiting_feedback",
  "detail": "Calories 45% of plan",
  "priority": 1,
  "action_url": "/curator/clients/:id"
}
```

Priority order:
1. Red KBZHU alerts (priority 1)
2. Overdue tasks (priority 2)
3. No activity 2+ days (priority 3)
4. Unread client messages (priority 4)
5. Weekly report without feedback (priority 5)

One client can appear multiple times with different reasons. Limit: 20 items.

### Extended Client List Fields

`GET /api/v1/curator/clients` extended with:
- `weekly_kbzhu_percent` — avg KBZHU completion % for the week
- `active_tasks_count` — active tasks count
- `overdue_tasks_count` — overdue count
- `last_activity_date` — last food entry date
- `streak_days` — consecutive days with food entries

### Benchmark Display in Hub

Below summary metrics — collapsible "Dynamics" block:
- Line chart: own weekly metrics vs platform benchmark
- Period: last 4/8/12 weeks (toggle)
- Metrics: KBZHU %, response time, task completion %, feedback %
- Color zone: above benchmark — green, below — red

### Visibility

- Curator sees only own data + anonymous benchmark
- Admin sees all curators' data (for future admin dashboard, out of scope)

## Section 6: Navigation and UX

### Route Structure

```
/curator                          — Analytics hub (new)
/curator/clients                  — Full client list (exists, enhanced)
/curator/clients/:id              — Client card with tabs (exists, enhanced)
/curator/clients/:id?tab=plan     — Plan tab
/curator/clients/:id?tab=tasks    — Tasks tab
/curator/clients/:id?tab=reports  — Reports tab
/curator/chat                     — Chat list (unchanged)
/curator/chat/:clientId           — Chat with client (unchanged)
/curator/content                  — Content management (unchanged)
/curator/content/new              — Create article (unchanged)
/curator/content/:id/edit         — Edit article (unchanged)
```

Tabs via query parameter `?tab=` — keeps routing simple, allows sharing links to specific tabs.

### Footer Navigation

Current: **Clients** | **Chats** | **Content** | **Profile**

Changed to: **Hub** | **Chats** | **Content** | **Profile**

"Hub" leads to `/curator` (analytics dashboard). Full client list accessible from hub's bottom block.

### Curator UX Flows

**Morning scenario:**
1. Open hub → see summary metrics and who needs attention
2. Click client from priority list → client card, Overview tab
3. See problem → switch to Plan or Tasks tab → adjust
4. Go to chat with client → discuss

**Sunday scenario (feedback):**
1. Hub shows "Awaiting feedback: N"
2. Click → client card → Reports tab
3. Review auto-collected week data
4. Fill structured feedback
5. Client receives notification

**Weekly scenario (planning):**
1. Hub → "Dynamics" block → see trends
2. Navigate to clients needing plan updates
3. Card → Plan tab → see auto values → adjust if needed

### Mobile UX

- Client card tabs: horizontal scroll tabs
- Create plan/task/feedback forms: full-screen modals (bottom sheet pattern), not inline
- Hub charts: horizontal scroll on narrow screens
- Priority list: vertical scroll with pull-to-refresh
