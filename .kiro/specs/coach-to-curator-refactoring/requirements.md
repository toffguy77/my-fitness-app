# Документ требований

## Введение

Рефакторинг терминологии в проекте BURCEV: замена роли "тренер" (coach) на "куратор" (curator) во всех компонентах системы. Куратор — это персональный фитнес-тренер и нутрициолог, который ведёт пользователя к его целям.

## Глоссарий

- **Система**: Приложение BURCEV для отслеживания питания и фитнеса
- **Куратор (Curator)**: Персональный фитнес-тренер и нутрициолог, ведущий клиента
- **Клиент (Client)**: Пользователь, работающий с куратором
- **Миграция_БД**: SQL-скрипт для изменения структуры базы данных
- **RLS_Политика**: Row Level Security политика в PostgreSQL

## Требования

### Требование 1: Рефакторинг Backend кода (Go)

**User Story:** Как разработчик, я хочу, чтобы терминология в Go-коде соответствовала бизнес-терминологии "куратор", чтобы код был понятен и согласован с документацией.

#### Критерии приёмки

1. WHEN переменная содержит "coach" в названии, THEN Система SHALL переименовать её в "curator" (coachID → curatorID, CoachID → CuratorID)
2. WHEN структура содержит поле "CoachID", THEN Система SHALL переименовать его в "CuratorID" с сохранением JSON-тега "curator_id"
3. WHEN функция содержит "Coach" в названии, THEN Система SHALL переименовать её с использованием "Curator"
4. WHEN комментарий содержит "coach", THEN Система SHALL заменить его на "curator"
5. WHEN сообщение об ошибке содержит "coach", THEN Система SHALL заменить его на "curator" (на английском для внутренних ошибок)
6. WHEN роль пользователя проверяется как "coach", THEN Система SHALL проверять роль "curator"

### Требование 2: Миграция базы данных

**User Story:** Как администратор БД, я хочу, чтобы структура базы данных использовала терминологию "curator", чтобы схема соответствовала бизнес-модели.

#### Критерии приёмки

1. WHEN таблица называется "coach_client_relationships", THEN Система SHALL переименовать её в "curator_client_relationships"
2. WHEN колонка называется "coach_id", THEN Система SHALL переименовать её в "curator_id" во всех таблицах (weekly_plans, tasks, weekly_reports)
3. WHEN колонка называется "coach_feedback", THEN Система SHALL переименовать её в "curator_feedback"
4. WHEN индекс содержит "coach" в названии, THEN Система SHALL создать новый индекс с "curator" и удалить старый
5. WHEN RLS-политика содержит "Coach" в названии, THEN Система SHALL создать новую политику с "Curator" и удалить старую
6. WHEN миграция применяется, THEN Система SHALL сохранить все существующие данные без потерь
7. WHEN миграция откатывается, THEN Система SHALL восстановить исходную структуру с "coach"

### Требование 3: Рефакторинг Frontend кода (TypeScript)

**User Story:** Как frontend-разработчик, я хочу, чтобы TypeScript-типы использовали терминологию "curator", чтобы код был согласован с backend.

#### Критерии приёмки

1. WHEN интерфейс содержит поле "coachId", THEN Система SHALL переименовать его в "curatorId"
2. WHEN интерфейс содержит поле "coachFeedback", THEN Система SHALL переименовать его в "curatorFeedback"
3. WHEN Zod-схема содержит поле "coachId", THEN Система SHALL переименовать его в "curatorId"
4. WHEN комментарий содержит "coach", THEN Система SHALL заменить его на "curator"
5. WHEN тестовые данные содержат "coach", THEN Система SHALL заменить их на "curator"

### Требование 4: Обновление документации

**User Story:** Как пользователь документации, я хочу, чтобы вся документация использовала термин "куратор", чтобы терминология была единообразной.

#### Критерии приёмки

1. WHEN markdown-файл содержит "coach" или "тренер", THEN Система SHALL заменить на "curator" или "куратор" соответственно
2. WHEN steering-файл содержит "coach", THEN Система SHALL заменить на "curator"
3. WHEN спецификация содержит "coach", THEN Система SHALL заменить на "curator"
4. WHEN README содержит "coaching", THEN Система SHALL заменить на "curation" или эквивалент

### Требование 5: Обновление тестов

**User Story:** Как разработчик, я хочу, чтобы все тесты использовали новую терминологию и продолжали проходить, чтобы убедиться в корректности рефакторинга.

#### Критерии приёмки

1. WHEN unit-тест содержит "coach" в переменных или данных, THEN Система SHALL заменить на "curator"
2. WHEN property-тест содержит "coach" в генераторах или проверках, THEN Система SHALL заменить на "curator"
3. WHEN mock-объект содержит "coach" в методах, THEN Система SHALL заменить на "curator"
4. WHEN тест проверяет роль "coach", THEN Система SHALL проверять роль "curator"
5. WHEN все тесты выполняются после рефакторинга, THEN Система SHALL обеспечить их успешное прохождение

### Требование 6: Обновление API

**User Story:** Как разработчик API, я хочу, чтобы все API-эндпоинты использовали терминологию "curator", чтобы API был согласован с остальной системой.

#### Критерии приёмки

1. WHEN API возвращает JSON с информацией о кураторе, THEN Система SHALL использовать поле "curator_id" вместо "coach_id"
2. WHEN API принимает запрос с информацией о кураторе, THEN Система SHALL ожидать поле "curator_id"
